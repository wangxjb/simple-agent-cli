# P10: simple-cli vs Claude Code 差距分析

## 总览

simple-cli 已经实现了一个从零构建的 Agent 核心闭环，但与 Claude Code 这个"工业级 Agent 产品"相比，还有大量高阶特性缺失。这些差距并非缺陷——simple-cli 的目标是**学习 Agent 原理**，Claude Code 的目标是**生产级 AI 编程工具**。

按重要性递进，差距分为四个层次。

---

## 第一层：Agent 基础能力（已实现）

这部分 simple-cli 已具备，是 Agent 的 "最小可行闭环"。

| 能力 | simple-cli | Claude Code |
|------|-----------|-------------|
| LLM 调用 | 流式/非流式，OpenAI 协议 | 流式，Anthropic API |
| 多模型切换 | 配置文件切换 5 个提供商 | Anthropic 模型族 |
| 工具系统 | 可插拔，5 个内置工具 | 20+ 工具，权限分级 |
| Agent 范式 | ReAct + FC 可切换 | 类似 FC 的工业实现 |
| 流式输出 | 逐 token 打印 | 逐 token + 富文本 |
| 多轮记忆 | 内存级，跨轮会话 | 上下文窗口管理 |
| 会话持久化 | JSON 文件，多会话管理 | 内置自动保存 |
| Tab 补全 | readline 实现 | 完整 CLI 补全 |

---

## 第二层：上下文工程（缺失）

Claude Code 最核心的工程能力之一，也是 simple-cli 接下来最值得学习的领域。

### 2.1 上下文窗口管理

```
Claude Code:
  消息历史 → 跟踪 token 数量 → 接近上限时自动压缩
  ├── 保留最近 N 轮完整对话
  ├── 对早期内容生成智能摘要
  └── 确保不会超出模型上下文限制

simple-cli:
  消息历史 → 无限追加 → 超出上限 → 模型报错或截断
```

**为什么重要：** 长时间对话会积累几千条消息，模型有 context window 限制（DeepSeek 64K token）。不管理上下文，会话会突然中断。

**学习要点：**
- Token 计数（tiktoken 库）
- 滑动窗口策略
- LLM 生成摘要压缩历史
- 压缩阈值决策（什么时候触发压缩）

### 2.2 系统提示工程

```
Claude Code 的系统提示（数千字）定义了：
  ├── 输出风格（engineer-professional）
  ├── 行为规范（危险操作确认、路径处理标准）
  ├── 编程原则（SOLID、KISS、DRY、YAGNI）
  ├── Git 安全协议
  ├── 上下文管理策略
  └── 工具使用优先级

simple-cli 的系统提示：
  "你是一个能使用工具的助手"（一行）
```

**为什么重要：** 系统提示是 Agent 的"性格和纪律"。没有好的系统提示，Agent 会做出不可预测的行为。

### 2.3 CLAUDE.md 项目感知

```
Claude Code 启动时自动：
  1. 读取项目根目录的 CLAUDE.md
  2. 读取 ~/.claude/CLAUDE.md（全局指令）
  3. 加载 memory/ 目录中的记忆文件
  4. 将以上内容注入系统提示

效果：Agent "知道"自己在什么项目里、有什么规范、用户是谁
```

**simple-cli 当前：** 没有项目感知，每次对话从零开始理解项目。

## 第三层：记忆系统（缺失）

Claude Code 有四类持久化记忆，跨会话保留。

### 3.1 记忆类型

| 类型 | 内容 | 生命周期 |
|------|------|----------|
| **user** | 用户角色、偏好、知识背景 | 长期 |
| **feedback** | 用户纠正的行为准则 | 长期 |
| **project** | 项目目标、决策、进度 | 中期 |
| **reference** | 外部资源指针（Slack、Linear 等） | 长期 |

### 3.2 与会话持久化的区别

```
会话持久化（simple-cli 已有）：
  "上次聊了什么" → 对话历史 → 当前项目目录

记忆系统（Claude Code）：
  "你是谁，你喜欢什么风格，这个项目的目标是..."
  → 跨会话、跨目录保留
  → 每次对话自动注入
```

**学习要点：**
- 结构化记忆模型（类型 × 内容 × 生命周期）
- 记忆索引与搜索
- 记忆更新/过期策略
- 与系统提示的整合

## 第四层：高阶 Agent 能力（缺失）

### 4.1 子代理（Sub-agents）

```
Claude Code 可以：
  Agent.run("搜索所有使用 React 的组件")  ← 主 Agent
    └── 自动创建子 Agent → 上下文隔离 → 返回摘要

效果：
  - 主 Agent 上下文不膨胀
  - 子 Agent 可以并行执行
  - 子 Agent 可以有不同的工具集
```

**simple-cli 当前：** 只有一个 Agent 实例，不能创建子代理。

### 4.2 计划模式（Plan Mode）

```
Claude Code 在复杂任务前：
  1. 声明进入"计划模式"
  2. 探索代码库
  3. 设计实现方案
  4. 生成书面计划
  5. 等待用户审批
  6. 用户批准后才开始写代码

效果：
  - 避免盲目实现
  - 用户对方案有控制权
  - 复杂任务不会被做偏
```

### 4.3 Task/Todo 系统

```
Claude Code 的 TaskCreate/TaskUpdate：
  - 将复杂任务分解为子任务
  - 跟踪每个子任务的完成状态
  - 支持依赖关系（blockedBy / blocks）
  - 实时反馈进度

效果：
  - 用户知道 Agent 在做什么
  - Agent 不会遗漏步骤
  - 长时间任务可追踪
```

### 4.4 工具输出截断

```
Claude Code 的 run_command 工具：
  命令输出 → 超过 2000 字符 → 自动截断 → 保留头尾

simple-cli 的 run_command 工具：
  已实现基本截断（max_output_chars=2000）
```

**基本已实现，但可以完善截断策略（保留头尾、显示截断位置）。**

### 4.5 MCP 协议支持

```
MCP (Model Context Protocol)：
  Agent ←→ MCP Server ←→ 外部能力
                              ├── 数据库
                              ├── API
                              └── 文件系统

效果：
  - 工具不再是代码内置的
  - 通过标准化协议连接外部服务
  - 生态可扩展
```

### 4.6 权限系统

```
Claude Code 的工具权限：
  ├── 自动允许（只读工具：Read, Glob, Grep）
  ├── 每次询问（写操作：Write, Edit, Bash）
  └── 禁止（rm -rf, force push）

simple-cli：
  所有工具无限制，直接执行
```

**simple-cli 学习环境可以不加限制，但了解权限模型是 Agent 工程的重要部分。**

### 4.7 Skills 系统

```
Claude Code 的 Skill 机制：
  用户: "帮我做 PPT"
    → 检测到 pptx skill
    → 加载 pptx skill 的指令和工具
    → Agent 获得 PPT 创建能力

效果：
  - 按需加载专业能力
  - 不占用常驻 token
  - 可社区贡献
```
